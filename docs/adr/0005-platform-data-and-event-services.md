# ADR 0005: Platform data and event services

- Status: Accepted
- Date: 2026-07-22

## Context

UWO products need shared object metadata, reliable notification delivery, privacy-safe operational analytics, and durable audit evidence. These capabilities must preserve the Phase 3A identity boundary and Phase 3B transactional/idempotency standards without selecting cloud storage, messaging vendors, brokers, or production databases. Prompts, outputs, file bytes, message bodies, credentials, and payment data must not leak into cross-service events.

## Decision

Create separate `platform_storage`, `platform_notifications`, `platform_analytics`, and `platform_audit` services with schema-versioned canonical contracts in `packages/contracts`. Each service owns repository protocols and a UnitOfWork. Committed thread-safe in-memory repositories implement rollback for tests only; production startup refuses to instantiate them.

Storage is metadata-only. Raw bytes remain behind an injected `BlobStore`; callers never choose storage keys. Object versions append immutably and bind content length, SHA-2 digest, tenant, product, region, and creator. Malware results form a separate append-only history; deterministic current scan status is derived without rewriting an object version. Scanner source IDs are globally scoped to the verified scanner executor and fingerprint tenant, object version, and status. Exact retry returns the immutable original result; conflicting reuse fails closed; result, idempotency record, and malware-detected event commit in one UnitOfWork. Expiry and retention are compared as parsed UTC instants. Restricted/regulated retention must be future-dated, and shortening requires a reasoned, audited platform-admin override. Download authorization contains an opaque token with bounded positive TTL, not a provider URL or credential, and fails closed when region, retention, legal-hold, scan, or integrity policy is unavailable.

Notification templates are immutable versions, numbered from the maximum existing version. Creation of an eligible notification and its delivery outbox record commits atomically after template product, region, and channel checks. Delivery follows claim-deliver-finalize: claim and lease transactionally, commit, call the provider outside the UnitOfWork with a stable idempotency key, then transactionally record attempt and terminal/retry state. Generic claim never independently dead-letters an owning notification. If the final claimed lease expires, reconciliation makes no provider call and atomically records the final attempt, creates exactly one dead letter, and terminalizes notification and outbox together. Repeated workers replay the aligned terminal state. Conflicting deduplication reuse fails closed and bounded exponential retry has no sleep dependency.

Analytics events are append-only and globally unique. Their contract contains only tenant, product, region, an allowlisted operational event type, fixed outcome/bucket/error dimensions, UTC time, and optionally an approved pseudonymous subject. Arbitrary metadata and sensitive content have no field. The service replaces caller `recorded_at`, enforces lateness and future-skew limits, and uses parsed half-open `[start, end)` UTC windows. Metric identities bind all dimensions and boundaries. Immutable snapshot hashes preserve reproducibility; minimum-count suppression applies to every export, and platform-admin cross-tenant output contains aggregate points only.

Audit uses append-only tenant streams with transactionally allocated monotonic sequence numbers. Every current SHA-256 hash binds the prior hash and canonical allowlisted scalar event content. Checkpoints verify the complete contiguous chain from genesis and reject missing/duplicate sequence positions. Range export verifies the full chain; offline verification rejects tenant substitution, noncontiguous or mismatched boundaries, broken previous/current hashes, and canonical digest mismatch. Source-event IDs and fingerprints make audit publishing idempotent and conflicting reuse fail closed. `actor_subject` always records the verified service executor. Pseudonymous source provenance remains a separate allowlisted attribute and is never promoted into authenticated actor identity. Audit events never retain request bodies, content, credentials, stack traces, or arbitrary exceptions.

Use provider-neutral deterministic `PlatformEvent` records to integrate the gateway, control plane, billing, and Phase 3C services. Event validation requires stable identifiers, UTC, a supported schema, deeply immutable allowlisted scalar attributes, and an ID fingerprint covering type, tenant, request, and safe attributes. Where a business mutation creates downstream work, including a tenant-status change, the immutable outbox record shares its transaction. Dispatchers take exclusive expiring leases, enforce due times and maximum attempts, recover abandoned work, preserve event identity, and acknowledge only after idempotent downstream acceptance. Duplicate acknowledgement is idempotent; cancellation and terminal business state terminally resolve work; poison events become visible dead letters.

Provider execution keeps an injected durable outcome repository and transactional outbox behind one UnitOfWork. It records the redacted provider result required to resume billing capture. After billing capture succeeds, capture state and mandatory `provider.execution.succeeded` insertion commit atomically. If this second transaction rolls back, retry reuses the provider result, calls billing capture idempotently, and enqueues the missing event without another provider call, charge, usage record, or event. Captured replay verifies or repairs the mandatory outbox record before returning. Provider-failure and compensation-failure states/events use the same restart-safe model; optional telemetry is isolated and cannot substitute for mandatory events. The committed implementation is in-memory for tests; production must supply durable workflow/outbox storage.

Application services own successful mutation and rollback audit events; rollback is emitted only after mutation begins. HTTP boundaries own exactly one rejected-request event and one unexpected-internal-error event, map provider/infrastructure failures separately from policy denials, and return only stable redacted errors.

## Consequences

Tests can prove idempotent/atomic malware ingestion, final-lease notification reconciliation without redelivery, aggregation privacy, verified audit actor provenance, complete export-chain validation, transactional rollback, lease exclusion, restart dispatch, and provider/capture/event deduplication without credentials or live providers. Production enablement still requires regional durable repositories, workflow outcomes, outboxes and blob storage, encryption and workload identity, approved malware/retention/legal-hold systems, real notification adapters, dispatch/reconciliation workers, integrity monitoring, backup/recovery, and Phase 3D telemetry/SLO/runbooks. This decision does not deploy infrastructure or store sensitive content in platform events, analytics, or audit.

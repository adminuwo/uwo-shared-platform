# ADR 0004: Billing credits and usage ledger

- Status: Accepted
- Date: 2026-07-21

## Context

Provider execution needs a tenant-isolated credit decision before work begins, a reservation before external cost is incurred, and a final usage charge after a safe response is accepted. Billing records must survive retries, concurrency, later price changes, and operational investigation without storing tenant prompts, model output, credentials, or payment data. Phase 3B must establish these invariants without selecting a production database or payment processor.

## Decision

Implement billing in the separate `services/platform_billing` boundary and keep its canonical schema-versioned contracts in `packages/contracts`. One credit equals 1,000,000 credit microunits. All credit amounts, rates, tokens, and usage quantities are integers; floating-point arithmetic is prohibited. Token rates are expressed per 1,000 tokens. Each input and output component is calculated independently as `ceil(tokens × rate / 1000)`, then the integer fixed request charge is added. This explicit round-toward-positive-infinity rule prevents systematic undercharging and is deterministic across runtimes.

Maintain an immutable append-only ledger. Grants and refunds increase available credit; adjustments change available credit by exactly their signed amount and cannot affect reserved credit; reservations move equal credit from available to reserved; captures decrease reserved credit; releases move equal credit from reserved to available. Contracts and repositories enforce these exact delta semantics, account/tenant binding, unique entry IDs, and sequential versions. Balances are derived from the ordered ledger and checked atomically: neither available nor reserved credit can become negative. Entries have no update or delete contract. A normal gateway finalization may append a capture entry and a release entry for unused reservation credit in one transaction.

Reservations move from pending or active to partially captured, captured, or released. A partial capture may later be captured again or release its remaining amount. Captured plus released credit can never exceed the original reservation. An expired reservation cannot be captured without an explicit platform-administrator override. Public billing API mutations retain explicit optimistic versions. Gateway receipts deliberately carry identity only: gateway-specific reserve, capture, and release operations load the current reservation and balance inside one UnitOfWork. This prevents unrelated ledger activity, concurrent reservations, or adapter recreation from applying stale creation-time versions. Normal gateway capture atomically records usage and releases unused credit; provider, missing-usage, or output-safety failure invokes an idempotent full release.

Store immutable rate-card versions keyed by product, shared UWO model alias, provider, and region. Repositories expose `active_at(as_of_utc)` and select only versions with `effective_at <= as_of_utc`; exact activation time is inclusive and future versions never activate early. Multiple versions in one rate-card family with the same effective time are rejected. Across families, `(effective_at, rate_card_id, version)` is the deterministic order. Capture evaluates at the usage occurrence time supplied by the application clock. Usage binds to the exact rate-card ID and version used for calculation, so a later card cannot alter historical charges. Committed prices are clearly marked test/example data and are not commercial provider prices.

Use one UnitOfWork for each credit grant and ledger entry, reservation and ledger entry, capture plus usage plus ledger entry, release and ledger entry, and refund and ledger entry. Test repositories are thread-safe, snapshot every participant, and roll back on injected failure. Production repositories must provide equivalent atomicity. Scope idempotency by operation, tenant, actor, and caller key; persist immutable original result snapshots. Exact replay returns the original result and emits no duplicate ledger, usage, or success-audit event. Different input under the same scope and key fails with a conflict.

Compose authorization with the identity and tenancy control plane. Explicit platform administrators manage accounts and credits. Active tenant members require `billing.read` to read their tenant. Explicitly configured, directory-revalidated internal executors may reserve, capture, and release. Tenant IDs select resources but never confer authority. Unknown or suspended tenants and suspended or closed billing accounts fail closed for debit activity. The billing translation boundary catches only canonical control-plane authorization and not-found errors. Unexpected identity, membership, role, or tenant repository failures propagate to the HTTP owner, which emits one redacted internal-error event and returns stable `500 internal_error`; it never misclassifies them as caller denials.

Expose a provider-neutral gateway lifecycle: authorize an estimated charge, reserve before provider execution, capture redacted token usage after output safety passes, and release on failure. Provider usage presence is mandatory and explicit. Missing or null usage has stable code `missing_usage`; partial, negative, or inconsistent counts are malformed. These cases fail before customer output, create no usage event, never fabricate zero counts, and compensate the reservation. Explicit zero input/output/total counts are valid and still pass through configured fixed-charge calculation.

Compensation is idempotent. If provider or output-safety failure occurs and release also fails, emit one redacted `billing-compensation-failed` event, return the stable non-sensitive `billing_compensation_failed` code, retain the original failure as an internal cause, and leave the reservation discoverable for recovery. If capture fails after a provider has succeeded, keep the completed provider result in a same-process pending-capture registry so an exact request retry performs capture without a second provider call. Production must replace this bootstrap registry with durable workflow/outbox state and reconciliation. Billing usage may contain tenant, product, model aliases, provider/model identifiers where allowed, region, correlation identifiers, integer token counts, rate-card version, and calculated charge. It must never contain prompts, model output, bearer tokens, API keys, provider secrets, request bodies, or payment credentials.

Application services own mutation-success, rollback, and compensation events; HTTP boundaries own authentication, authorization, not-found, and unexpected-error events. Emit a transaction-rollback event only when a repository failure occurs after the first mutation begins. Validation errors, stale versions, idempotency conflicts, and denials are not transaction rollbacks. Each denied HTTP request and each unexpected HTTP failure produces exactly one owning event.

## Consequences

Credit state is reproducible from semantically validated immutable entries, concurrent reservations cannot overdraw a tenant, unrelated financial activity cannot stale a gateway lifecycle, historical pricing remains stable, missing provider usage cannot become a free usage record, and exact retries do not repeat provider work or double-charge. Service and gateway tests require no live provider or credential. The in-memory repositories, in-process gateway adapter, pending-capture registry, example pricing, and JSON audit sink are test/bootstrap implementations only. Production enablement still requires durable regional transactional and workflow storage, workload identity, immutable audit ingestion, rate-card approval, reconciliation, expiration workers, monitoring, and recovery procedures. This decision does not connect a payment gateway or deploy infrastructure.
